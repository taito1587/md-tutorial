# Step 04 — Production MD (本番シミュレーション)

> このステップで必要な前知識: [PRIMER](../../PRIMER.md), Step 03 が完了していること

## このステップで何をやるか

加熱・平衡化の終わった系を、**300 K, 1 atm の条件下で 10 ns の MD** に乗せます。これがシミュレーションの「本番」 — ここで取れたトラジェクトリが研究の対象データになります。

具体的なアウトプット:

- `03_Prod.ncrst` — 最終時点 (10 ns 後) の座標 + 速度
- `03_Prod.nc` — トラジェクトリ (50,000 フレーム ≒ 10 ns 分)
- `03_Prod.out` — 全エネルギー・温度・圧力・密度のログ

> ⚠️ **このステップが一番時間がかかります**。CPU シリアルだと公式は ~13 時間と言っています。学習用ならまず短縮版 (1 ns or 100 ps) で動作確認することを強く推奨します。

## 2. NPT アンサンブルに切り替える理由

### NVT (体積一定) → NPT (圧力一定)

| | NVT (Step 03) | NPT (Step 04) |
|---|---|---|
| 何が一定? | 粒子数 N、体積 V、温度 T | 粒子数 N、**圧力 P**、温度 T |
| 箱は? | 固定 | **伸縮できる** |
| 用途 | 加熱中、表面のあるシステム | **本番 MD の定番** |

### なぜ NPT なのか

**実験は通常 1 気圧の環境**で行われます。シミュレーションも実験条件を再現するため、圧力を 1 atm に制御するのが自然です。

NPT では系の体積が時々刻々と少し伸縮しながら平均圧力 = 1 atm を実現します。これにより:

- 密度が**実験的な値** (水なら ≒ 1.0 g/cm³) に自動的に落ち着く
- 異なる温度・組成でも実験と直接比較できる

## 3. Berendsen barostat (`barostat=1`) — 圧力を制御する仕組み

### 何をするものか

各ステップで瞬間圧力 `P_inst` を計算し、目標圧力 `P_0` (= 1 atm) との差に応じて **箱と原子座標を一律スケール** します:

```
λ = (1 - τ_p / Δt × κ × (P_0 - P_inst))^(1/3)

新しい箱の辺 = λ × 古い箱の辺
新しい原子座標 = λ × 古い原子座標
```

(τ_p は緩和時間、κ は圧縮率)

直感的には:

- 圧力が高すぎる → 箱を少し膨らませる
- 圧力が低すぎる → 箱を少し縮める

を毎ステップ繰り返すことで、平均圧力を 1 atm に収束させる仕組み。

### Berendsen の長所と短所

| 長所 | 短所 |
|------|------|
| 実装が単純で**安定** | **「正しい統計集団」にはならない** (圧力ゆらぎが理論値より小さい) |
| **平均値**は正しい (温度・密度の平均) | 圧力**揺らぎ**が必要な解析 (圧縮率測定など) には使えない |

このチュートリアルや教育目的では Berendsen で十分ですが、**最先端の研究では Monte Carlo barostat (`barostat=2`)** がより推奨されます (公式 AMBER もそちらを推奨)。チュートリアル原文に倣ってここでは `barostat=1` を使います。

## 4. リスタートの仕組み — `irest=1`, `ntx=5`

加熱で得た `02_Heat.ncrst` には**座標 + 速度 + 時間カウンタ**が全て入っています。これを引き継ぐ設定:

| パラメータ | 値 | 意味 |
|-----------|----|------|
| `ntx` | `5` | 座標 **+** 速度を読む (3 や 1 だと座標のみ) |
| `irest` | `1` | リスタート: 時間カウンタ・ステップ番号を引き継ぐ |

これにより、production 開始時点で:
- 各原子は加熱で得た 300 K 相当の速度を持っている
- 出力ログの `TIME(PS)` は加熱からの累積時間でカウントされる (連続した 1 つの軌跡として扱える)

新規実行 (`irest=0`, `ntx=1`) との違い:

```
irest=0, ntx=1     ────► 速度を再アサイン (= 系がリセットされる)
irest=1, ntx=5     ────► 加熱の状態をそのまま継承 (= スムーズに本番突入)
```

リスタートする方が**平衡が崩れない**ので、production には必須の設定です。

## 5. なぜ `nstlim=5000000` なのか

### 計算

```
nstlim × dt = 全シミュレーション時間
5,000,000 × 0.002 ps = 10,000 ps = 10 ns
```

### アラニンジペプチドにとっての 10 ns

| 時間スケール | 何が観察できるか |
|------------|---------------|
| 100 ps | 局所的なゆらぎ、結合の振動 |
| 1 ns | 側鎖の回転 |
| **10 ns** | **φ/ψ 二面角の遷移を数回サンプリング** |
| 100 ns | アラニンの (φ, ψ) 分布が収束 |
| μs〜ms | フォールディング (タンパク質の場合) |

10 ns は「アラニンの主要な構造を一通り観察できる」「ただし収束には少し足りないかも」くらいの長さ。教育用としては適切ですが、本格的な研究では 100 ns 以上が普通です。

### 短縮版

学習目的なら **1 ns (`nstlim=500000`) や 100 ps (`nstlim=50000`)** でパイプラインを通すのが賢明です。後で本番を走らせる時間を取れば良い。

## 7. 実行してみる

```bash
cd workspace
mkdir -p 04_production && cd 04_production

cp ../../solutions/04_production/03_Prod.in .

# 動作確認なら nstlim を減らす (10 ns → 1 ns)
# sed -i.bak 's/nstlim   = 5000000/nstlim   = 500000/' 03_Prod.in

# pmemd は AmberTools にも入っている高速エンジン (CPU シリアル版)
# GPU/MPI 並列の高速版は商用 AMBER のみ
pmemd -O \
    -i 03_Prod.in \
    -o 03_Prod.out \
    -p ../01_setup/diala.parm7 \
    -c ../03_heating/02_Heat.ncrst \
    -r 03_Prod.ncrst \
    -x 03_Prod.nc \
    -inf 03_Prod.mdinfo &

echo "Started production MD (PID $!)."
echo "Monitor: tail -f 03_Prod.out"
echo "Or:      cat 03_Prod.mdinfo  # current step + ETA"
```

末尾の `&` でバックグラウンド実行になります。ターミナルを閉じても止まらないようにするには `nohup` をつけます:

```bash
nohup pmemd -O ... &
```

### `sander` で代用する場合

`pmemd` がうまく動かない場合 (Nix で見つからない等) は、同じ入力ファイルで `sander` でも実行できます:

```bash
sander -O -i 03_Prod.in ... &
```

`sander` の方が遅いので、所要時間は数倍に延びます。

## 9. 出力の見方

```
04_production/
├── 03_Prod.in       # 入力
├── 03_Prod.out      # ★エネルギー・温度・圧力・密度のログ
├── 03_Prod.ncrst    # 10 ns 後の座標 + 速度 (リスタート用)
├── 03_Prod.nc       # ★トラジェクトリ (50,000 フレーム ≒ 10 ns)
└── 03_Prod.mdinfo   # 進捗情報
```

### `03_Prod.out` の追加項目

NPT なので、heating ではほぼ意味のなかった量が意味を持ちます:

```
 NSTEP =     100   TIME(PS) =    20.200   TEMP(K) =  300.45   PRESS =   1.23
 Etot   = ...     EKtot   = ...    EPtot   = ...
 BOND   = ...     ANGLE   = ...    DIHED   = ...
 1-4 NB = ...     1-4 EEL = ...    VDWAALS = ...
 EELEC  = ...     EHBOND  = 0.0    RESTRAINT = 0.0
 EKCMT  = ...     VIRIAL  = ...    VOLUME  = 35198.34
 Density   =        0.998
```

| 項目 | 意味 | 期待される値 |
|------|------|------------|
| `TEMP(K)` | 瞬間温度 | 300 ± 5 K (Langevin が制御) |
| `PRESS` | 瞬間圧力 (atm) | 1 atm 周辺で大きく振動 (-100 〜 +100 atm が普通) |
| `VOLUME` | 系の体積 (Å³) | 平衡後はほぼ一定 |
| `Density` | 密度 (g/cm³) | **約 1.0** (水ベース) |
| `EKCMT`, `VIRIAL` | 圧力計算用の中間量 | (確認は不要) |

> 💡 **PRESS の瞬間値は大きく揺れる**ので「-50 atm が出た」と慌てなくて OK。 平均で 1 atm 付近にいれば正常。後で `process_mdout.perl` で平均を取る。

### 健全な production の指標

```bash
# 温度の平均と標準偏差
grep "TIME(PS)" 03_Prod.out | awk '{sum+=$9; n++} END {print "mean TEMP =", sum/n}'

# 密度の推移
grep "Density" 03_Prod.out | awk '{print NR, $3}' | head -20
```

期待される結果:

- ✅ 平均温度: **300 ± 1 K** 程度
- ✅ 平均密度: **0.99〜1.00 g/cm³**
- ✅ Etot がドリフトしていない (緩やかな上昇/下降は OK だが急激な変動はダメ)
- ✅ 体積が平衡値の周りで揺らいでいる

## 11. つまずきポイント

| 症状 | 原因と対処 |
|------|-----------|
| シミュレーションが途中で止まる | ディスク容量不足。`*.nc` は 10 ns で 数百 MB 〜 数 GB |
| `vlimit exceeded` | 平衡化が不十分。Heating の後に短い NPT 平衡化を挟む (このチュートリアルでは省略) |
| 異常な圧力スパイク | NPT 直後の典型現象。最初の数 ps を捨てるのが普通 |
| 密度が 1.0 から大きく外れる | 力場 + 水モデルの組み合わせミス。ff19SB は OPC と組むこと |
| ログに `TIME(PS)` が増えていかない | 計算がハングしている。`top` で CPU 使用率確認 |
